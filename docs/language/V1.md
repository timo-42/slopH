# SlopH Language v1

Status: normative implementation profile.

This document fixes the v1 boundary. Detailed numeric encodings and the public
Core serialization remain in their dedicated normative documents; research and
rationale documents are not part of the language contract.

Every Source v1 project implicitly receives the mandatory `core` library.
Other compiler-bundled libraries are explicit manifest dependencies, for
example `dependencies = ["std"]`. A bundled package may define its package-root
module in `src/root.sloph`; the core library uses this convention to define the
ordinary nominal `core::Bool`, `core::Unit`, `core::Option`, and `core::Result`
types and the Boolean operations.
Descriptor I/O is supplied by the ordinary `syscall` and `std::io` libraries
described in
[the POSIX boundary](../toolchain/POSIX_BOUNDARY.md), not by a Core `io.write`
primitive.

The bundled `std::bytes` module exposes `length(Bytes) -> Int` and
`is_empty(Bytes) -> Bool`. These functions keep the `bytes.length` primitive
behind an ordinary library API.

The bundled `std::int` module exposes `to_bytes(Int) -> Bytes`. It returns the
canonical base-10 ASCII representation: zero is `"0"`, negative values have
one leading `-`, and no other value has a leading zero. The function keeps the
`int.to_bytes` primitive behind an ordinary library API.

The optional top-level `math` package is independent of `std`. Its
`math::int` module exposes `absolute`, `minimum`, `maximum`, `sign`, `distance`,
`is_negative`, and `is_positive` for unbounded `Int` values. Projects opt in
with `dependencies = ["math"]`.

## Execution and Programs

Evaluation is strict and left-to-right. A delayed computation is an explicit
owned closure. Source modules have no runtime initialization: top-level values
are bounded, pure compile-time constants.

On a command host, an executable explicitly depends on `os`, imports
`os::process::Exit`, and exports exactly one entry function:

```text
import os::process::{Exit}

public fn main() -> Exit effects {io, nondeterministic} {
    ...
}
```

The command entry contract requires a host process capability. Native command
targets provide it; future WASI CLI and Node/Bun command runners may provide
equivalent adapters. Browser-hosted WebAssembly has no process exit and exposes
callable functions instead of this command entry contract.

The declared effect set may omit effects not used by the body. Process
arguments, streams, files, environment, clocks, randomness, and blocking
networking are accessed through effect-tracked global standard-library APIs.
V1 has no threads, tasks, channels, async execution, or runtime global values.

Recoverable failure uses `Result`. The standard `?` transformation propagates
an error. A trap aborts the process without unwinding. `defer` actions run in
last-in-first-out order on ordinary scope exits and explicit control transfer,
including `Result` propagation, but do not run after a trap.

## Values, Functions, and Patterns

Functions are recursive, higher-order, and may be partially applied. A
zero-argument source function is represented as a function accepting `Unit`.
Closures own their environments: copyable captures copy, other captures move,
and borrowed captures cannot escape their call.

Function declarations may use one body or ordered pattern clauses:

```text
fn factorial(n: Int) -> Int
| 0 => 1
| n => n * factorial(n - 1)
```

Clauses are tried from top to bottom. Patterns support resolved constructors,
single-constructor structs, tuples, binders, wildcards, nesting, and exact
`Int` literals. Guards are evaluated only after their pattern matches. A
guarded clause does not establish exhaustiveness; every declaration requires
an unguarded structurally exhaustive path. Provably unreachable clauses are
errors.

Local immutable bindings use `let`. `var`, assignment, `while`, `loop`,
`break`, `continue`, and `return` are source forms lowered before Core
optimization. Their evaluation order remains observable when effects occur.

## Types and Ownership

Core provides exact unbounded `Int` and the standard parameterized
`Float[Format, Behavior]` family. Nominal bounded integer and floating types
follow [INTEGER.md](./INTEGER.md) and [FLOAT.md](./FLOAT.md). Conversions are
explicit. There is no general subtyping or implicit numeric coercion.

Enums are nominal. Structs are single-constructor enums. `Bool`, `Unit`,
`Option`, `Result`, and tuples are mandatory-library nominal types using the
same constructor and exhaustive-case semantics as user types.

Generic type and function parameters are explicit in declarations. Every use
of a generic type, function, or constructor supplies all type arguments; V1
does not infer them. Case patterns omit type arguments because the scrutinee
already fixes the instantiated enum type:

```text
type Box[Item] { Box(item: Item); }
fn wrap[Item](item: Item) -> Box[Item] { Box::Box[Item](item) }
let box: Box[Int] = wrap[Int](42);
case box -> Int { Box::Box(item) => { item } }
```

V1 has no higher-kinded parameters, GADTs, type
families, dependent types, variance, variadic generics, or general constant
generics.

The first implemented generic profile is unconstrained. A future `interface`
constraint may select one coherent `implementation`. An
implementation must be declared in the workspace package defining either the
interface or the implementing type, and at most one implementation may exist
for a type/interface pair. Core receives the selected implementation as an
ordinary explicit witness value.

Every non-copy value has one owner. Assignment and argument passing move it
unless the operation explicitly borrows or clones. Parameter modes are `own`,
`borrow`, and `borrow mut`. A borrow exists only for its call and cannot be
stored, returned, captured, or placed in a field. Destruction is deterministic,
pure, non-suspending, and infallible.

## Effects and Unsafe Code

Function types contain a finite subset of `io`, `nondeterministic`, and
`may_suspend`. Empty means pure. Exported functions declare an upper bound;
private bodies infer their effects. Higher-order declarations may bind one
finite effect variable, which is solved only by union and subset constraints.

Mutation through `borrow mut` is visible in the parameter type rather than an
effect bit. Trapping and unsafe provenance are checked Core metadata. Trusted
and foreign operations require an explicit `unsafe` scope; unsafe authority is
not forgeable through a transformation.

## Modules and Extensibility

A source file maps deterministically to one package-qualified module. Package
and module dependency graphs are acyclic. Declarations share one namespace;
constructors are selected through their nominal type. Ordinary, `transform`,
and `operator` imports are explicit, non-transitive, and independently
conflict-checked. V1 has no wildcard import or public re-export.

The compiler supplies the virtual, compile-time-only `compiler::target`
namespace. Its `platform` value has type `(OS, Arch)` and its `arch` value has
type `Arch`. Typed target constants use the lowercase `os` and `arch`
namespaces. The initial constants are `os::linux`, `os::darwin`,
`arch::amd64`, and `arch::arm64`. They may select imports before name and type
resolution using ordinary tuple-pattern structure:

```text
import case compiler::target::platform {
  (os::linux, arch::amd64) => syscall::posix::linux::amd64::{native_write_once};
  (os::darwin, arch::arm64) => syscall::posix::darwin::arm64::{native_write_once};
}
```

Exactly one branch must match. A module may constrain its own availability,
for example `module cpu::amd64 when compiler::target::arch is arch::amd64;`.
Importing an unavailable module is an error. Compiler target values are
build-graph inputs, not runtime values and not a mechanism for detecting
optional CPU instructions on the executing machine.

A declarative `transform` begins with a resolved name and expands hygienically
through typed syntax captures plus quote/splice. It is deterministic and has no
host access. Expansion depth, work, memory, and output size are bounded. V1 has
no arbitrary compile-time functions or build tasks.

ASCII infix operators declare precedence `0..9`, associativity `left`, `right`,
or `none`, and one binary interface method. Operators cannot introduce
short-circuiting or selective evaluation. Prefix, postfix, Unicode, and mixfix
operators are outside v1.

Source is UTF-8; identifiers and operators are ASCII. Line and block comments
are retained by the public Syntax representation. A source byte literal
constructs `core::Bytes`. V1 deliberately has no standard `Text` type.

## Core Boundary

Canonical typed Core has exactly these expression forms:

```text
Var Lit App Lam Let Case Type Prim Con
```

It is explicitly typed and uses resolved global identities. Type abstraction
and application use typed `Lam` and `App`. Enums, structs, and standard data use
`Con` and exhaustive `Case`. Surface mutation, clauses, guards, conditionals,
loops, error propagation, and operators introduce no additional Core form.

Elaborated Core retains ownership modes and effects. Before the stable
optimized boundary, ownership is lowered into explicit cataloged move, clone,
borrow, and drop operations. Both stages are independently validated.

The generated normative context bundle containing this specification, Core,
the primitive catalog, and mandatory `core` API must not exceed 131072 UTF-8
bytes.

## Supported Toolchain Profile

The production v1 compiler is Python 3.11+ and is exposed through one `sloph`
command. Native output is deterministic C11 compiled by the host toolchain on
macOS ARM64 or Linux AMD64. Compiler packages and libraries are local workspace
members. V1 has no registry, network dependency resolver, LSP, direct object
backend, standalone bundled Python executable, self-hosting proof, or managed
backend.
